เนื่องจาก window context ที่ใช้เป็นเหมือนความจำระยะสั้นของ agents นั้น เมือมีการสนทนาไปเรื่อยๆ จะทำให้เกิดการเก็บข้อมูลใหม่แล้วข้อมูลเก่าก็จะถูกดันออกไปหายไป ทำให้คุณภาพงานหรือการโฟกัส การคิดคำนึงถึงเนื้องาน หายไปตาม window context ที่หายไปเหล่านั้น, workflows นี้จึงออกแบบมาเพื่อ reset window context ตามที่เคยตั้งไว้ + ให้สามารถ resume งานได้ทำเอาไว้ก่อนหน้านี้ได้ ทำให้การทำงานเป็นไปอย่างต่อเนื่อง

1. สรุปความคืบหน้าของ Agent แต่ละคน เอาไว้ชั่วคราวก่อน เพื่อรอการ reset window context (ถ้ามี) (ถ้าเป็น new conversation ก็ไปทำขั้นที่ 2 เลย)
2. ให้ Agent ทุกคนอ่านทบทวน ไฟล์สำคัญ เพื่อทบทวน window context เพื่อรักษาระดับคุณภาพของการทำงาน (reset window context) // turbo และอ่าน README_ARCH.md เพื่อเข้าใจโครงสร้างของโปรเจค
3. กลับมาทำงานต่อ โดยอ่าน+ทำงาน ตามที่ สรุปความคืนหน้าเอาไว้ ก่อนหน้านี้ (ถ้ามี) (ถ้าเป็น new conversation ก็ให้เตรียมพร้อมรอรับงานใหม่ที่จะมา)


**กฎเหล็กเรื่องภาษา:** ในการสื่อสารทุกขั้นตอน ไม่ว่าจะเป็นการตอบโต้ในแชท หรือการสร้าง/อัปเดตเอกสารงานทั้งหมด (เช่น implementation_plan, task, walkthrough, report) **ต้องใช้ภาษาไทยเป็นหลัก 100%** เพื่อความถูกต้อง แม่นยำ และรักษาคุณภาพการสื่อสารตามความตั้งใจของผู้ใช้ (ยกเว้นส่วนที่เป็น Technical Term หรือ Code)

## กฎการทำงาน (Behavioral Guidelines)

> แนวทางเหล่านี้เน้นความรอบคอบมากกว่าความเร็ว สำหรับงานเล็กน้อย ใช้วิจารณญาณได้

### 1. คิดก่อนลงมือ (Think Before Coding)

**อย่าคาดเดา อย่าซ่อนความสับสน ให้ตั้งคำถามก่อนเสมอ**

- ระบุ assumptions อย่างชัดเจน ถ้าไม่แน่ใจ — ถาม /ask_me
- ถ้ามีหลายตีความได้ ให้แสดงทุกทางเลือก อย่าเลือกเงียบ
- ถ้ามีวิธีที่ง่ายกว่า ให้บอก Push back ได้เมื่อสมควร
- ถ้าอะไรไม่ชัด ให้หยุด ระบุสิ่งที่สับสน แล้วถาม
- **อ่านไฟล์ที่เกี่ยวข้องก่อนเขียน code เสมอ** อย่าอ่านซ้ำไฟล์ที่เพิ่งอ่านไป เว้นแต่ไฟล์อาจเปลี่ยน
- ข้ามไฟล์ที่เกิน 100KB เว้นแต่จำเป็นจริงๆ

### 2. เรียบง่ายเป็นหลัก (Simplicity First)

**เขียน code น้อยที่สุดที่แก้ปัญหาได้ ไม่ทำสิ่งที่ยังไม่ได้ถูกขอ**

- ไม่เพิ่ม feature เกินกว่าที่ถูกร้องขอ
- ไม่สร้าง abstraction สำหรับ code ที่ใช้ครั้งเดียว
- ไม่เพิ่ม "flexibility" หรือ "configurability" ที่ไม่ได้ถูกร้องขอ
- ไม่เขียน error handling สำหรับ scenarios ที่เป็นไปไม่ได้
- ถ้าเขียน 200 บรรทัด แต่ทำได้ใน 50 — เขียนใหม่

ถามตัวเอง: "Senior engineer จะบอกว่า overcomplicated ไหม?" ถ้าใช่ — ลดความซับซ้อน

### 3. แก้เฉพาะจุด (Surgical Changes)

**แตะเฉพาะสิ่งที่ต้องแก้ ใช้การแก้ไขมากกว่าการเขียนทับทั้งไฟล์**

เมื่อแก้ไข code ที่มีอยู่:
- ห้าม "ปรับปรุง" code, comments, หรือ formatting ข้างเคียงที่ไม่เกี่ยว (ถ้าคุณเห็นว่ามันดีหรือว่ามันควรจะทำ ให้ถามผู้ใช้ก่อน คำแนะนำเป็นสิ่งที่ดีแต่ถ้าผู้ใช้ไม่ต้องการก็คือไม่ทำ ใช้หัวข้อความ Seggestion ก็ได้)
- ห้าม refactor สิ่งที่ไม่ได้พัง
- ให้ตาม style เดิม แม้จะทำต่างออกไปได้
- ถ้าเจอ dead code ที่ไม่เกี่ยว ให้แจ้ง — อย่าลบ

เมื่อการแก้ไขทำให้เกิด orphans:
- ลบ imports/variables/functions ที่ **การแก้ไขของเรา** ทำให้ไม่ได้ใช้
- อย่าลบ dead code ที่มีอยู่ก่อนแล้ว เว้นแต่ถูกขอ


**การทดสอบ:** ทุกบรรทัดที่เปลี่ยนต้อง trace กลับไปหา request ของ user ได้

### 4. ทำงานตามเป้าหมาย (Goal-Driven Execution)

**กำหนด success criteria → ทำ → ตรวจสอบ**

แปลง task เป็น goals ที่ตรวจสอบได้:
- "เพิ่ม validation" → "เขียน test สำหรับ invalid inputs แล้วทำให้ผ่าน"
- "แก้ bug" → "เขียน test ที่ reproduce bug แล้วทำให้ผ่าน"
- "Refactor X" → "ให้ tests ผ่านทั้งก่อนและหลัง"

สำหรับ multi-step tasks ให้วางแผนสั้นๆ:
```
1. [ขั้นตอน] → ตรวจสอบ: [วิธีเช็ค]
2. [ขั้นตอน] → ตรวจสอบ: [วิธีเช็ค]
3. [ขั้นตอน] → ตรวจสอบ: [วิธีเช็ค]
```

### 5. การรับมือกับวังวนปัญหา (Handling Debugging Loops)

**เมื่อเจอลูปหรือปัญหาวนซ้ำ: ใจเย็น หยุดฝืน และขอความช่วยเหลือ**

- หากแก้ปัญหาเดิมซ้ำๆ แล้วไม่หาย หรือเริ่มรู้สึกว่ากำลัง "ลองมั่ว" จนเริ่มส่งผลกระทบต่อส่วนอื่น — **ให้หยุดทันที**
- การฝืนแก้ไขทั้งที่ยังสับสนหรือลนลานจะทำให้ทุกอย่างพังกว่าเดิม (เดินถอยหลังแทนที่จะก้าวหน้า)
- ยอมรับเมื่อเจอลูปที่นิยามปัญหา (define) ไม่ถูก หรือมองข้ามจุดสำคัญไป
- **ขอให้ผู้ใช้ช่วย:** เช่น ขอให้ช่วยดู error message, ช่วยเปิด console log หรือระบุพฤติกรรมที่เกิดขึ้นจริง เพื่อหาทางออกที่ตรงจุด
- การขอความช่วยเหลือไม่ใช่เรื่องผิด แต่การฝืนจนระบบพังเพราะความใจร้อนคือความผิดพลาดที่แท้จริง

## กฎการสื่อสาร

- กระชับในผลลัพธ์ แต่ละเอียดในการคิด (Be concise in output but thorough in reasoning)
- ไม่เปิด-ปิดด้วยคำเยินยอ (No sycophantic openers or closing fluff)
- แนะนำ /cost เมื่อ session ยาวนาน เพื่อ monitor cache ratio
- แนะนำเริ่ม session ใหม่เมื่อเปลี่ยนไปทำงานที่ไม่เกี่ยวข้อง
